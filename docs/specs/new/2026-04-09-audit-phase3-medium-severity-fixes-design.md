## Group 1: CLI Command Fixes

### Fix 1.1: `logs -f` Log Rotation Handling (§5.1)

**File:** `src/commands/logs.ts`

**Problems:**
- Lines 49-63: Follow mode only checks `if (newSize > lastSize)`. When a log file rotates (newSize < lastSize), nothing happens — reads produce stale or garbage content.
- Line 20: `parseInt(opts.lines, 10) || 50` treats `--lines 0` as falsy, returning 50.

**Required Changes:**

1. In the follow loop, add a rotation check: if `newSize < lastSize`, reset `lastSize` to 0 and re-read from the beginning of the new file.
2. Replace `|| 50` with a proper default: `const lineCount = opts.lines !== undefined ? parseInt(opts.lines, 10) : 50;` — validate that the parsed value is a non-negative integer.

**Tests:**
- Test that `--lines 0` returns 0 lines (not 50)
- Test that follow mode handles file truncation (newSize < lastSize) by resetting

---

### Fix 1.2: `status` NaN Guard (§5.2)

**File:** `src/commands/status.ts`

**Problem:** Lines 8-13: `formatElapsed` does not guard against invalid `startedAt` values. `new Date(undefined).getTime()` returns `NaN`.

**Required Changes:**

1. In `formatElapsed`, check if the parsed date is valid before computing elapsed time:
   ```typescript
   const start = new Date(startedAt).getTime();
   if (isNaN(start)) return "unknown";
   ```

**Tests:**
- Test `formatElapsed` with `undefined`, `null`, empty string, and invalid date string inputs
- All should return `"unknown"` instead of `NaN`

---

### Fix 1.3: `.env` Template Alignment (§13.1)

**File:** `src/commands/init.ts`

**Problem:** Lines 84-93: The `.env` template generated by `shkmn init` does not include `SLACK_WEBHOOK_URL`, but `shkmn doctor` checks for it in `REQUIRED_ENV_KEYS`.

**Required Changes:**

1. Add `SLACK_WEBHOOK_URL=` to the `.env` template in `init.ts`, with a comment: `# Optional — Slack webhook for notifications`
2. Alternatively, remove `SLACK_WEBHOOK_URL` from `REQUIRED_ENV_KEYS` in `doctor.ts` if it's truly optional.

**Decision:** Align init with doctor — add the key to the template. If the key is optional, downgrade the doctor check from error to warning.

**Tests:**
- Test that the generated `.env` template includes all keys that doctor checks

## Group 2: Pipeline Core Fixes

### Fix 2.1: Deferred Tasks Invisible in Status (§5.5)

**File:** `src/core/pipeline.ts`

**Problem:** Lines 391-399: `state` is declared inside a `try` block but referenced after the `catch` block, outside its scope. This is a scoping bug that would cause a `ReferenceError` at runtime.

**Required Changes:**

1. Move the `state` declaration outside the try block, or restructure to keep `activeRuns.set` inside the try block:
   ```typescript
   let state: RunState;
   try {
     state = readRunState(taskDir);
     if (state.status !== "running") continue;
   } catch {
     continue;
   }
   if (activeRuns.has(slug)) continue;
   activeRuns.set(slug, state);
   ```

**Tests:**
- Test that `retryDeferredTasks` adds tasks to `activeRuns`
- Test that `getActiveRuns()` includes retried deferred tasks
- Test that `shkmn status` shows deferred tasks after retry

### Fix 2.2: `shouldNotify` Dead Guard (§5.7)

**File:** `src/surfaces/types.ts`

**Problem:** Line 74: `case "stages": return true;` makes the guard in `console-notifier.ts` effectively dead code.

**Required Changes:**

1. Define a `STAGES_EVENTS` set that explicitly lists which event types the `"stages"` level should emit (all currently defined event types).
2. Replace `return true` with `return STAGES_EVENTS.has(event.type)`.
3. This makes the guard functional — future event types added without being registered in the set will be filtered by default.

**Tests:**
- Test that `shouldNotify("stages", event)` returns true for all current event types
- Test that `shouldNotify("stages", { type: "unknown_future_type" })` returns false
- Test that `shouldNotify("minimal", event)` still filters correctly

### Fix 2.3: Extract `processStage` to `stage-runner.ts` (§10.1)

**File:** `src/core/pipeline.ts`

**Problem:** `createPipeline` is 662 lines with `processStage` being 455 lines of nested closures. This makes the file hard to navigate, review, and test in isolation.

**Required Changes:**

1. Extract `processStage` and its helper functions into a new `src/core/stage-runner.ts` module.
2. `stage-runner.ts` exports a `runStage(ctx: StageContext): Promise<StageResult>` function.
3. Define a `StageContext` interface containing all the dependencies `processStage` currently captures via closure:
   - `runtimeDir`, `config`, `runner`, `emitNotify`, `activeRuns`, `registry`
   - `state`, `slug`, `taskDir`, `stage`
4. `createPipeline` calls `runStage(ctx)` instead of the inline `processStage`.
5. No behavioral changes — this is a pure refactor. All existing tests must pass without modification.
6. **Dependency:** This fix benefits from the `PipelineStage` type union fix (Phase 2, Fix 4) being in place first, so `StageContext` can use the complete type.

**Tests:**
- All existing `pipeline.test.ts` tests pass unchanged
- New unit tests for `runStage` can test stage logic in isolation (without the full pipeline factory)

## Group 3: Agent Prompt Safety

### Fix 3.1: `quick-execute.md` Guardrails (§8.3)

**File:** `agents/quick-execute.md`

**Problem:** Only 3 guardrails (no stages, no tests unless asked, be concise). No security constraints despite having full Read/Write/Edit/Bash/Web/Slack/Notion access.

**Required Changes:**

Add safety guardrails to the prompt:
- Do not access or output API keys, tokens, passwords, or credentials
- Do not run `git push --force` or delete branches
- Do not delete production resources or data
- Do not send Slack messages to channels other than the configured pipeline channel
- Do not modify Notion pages unless the task explicitly requires it
- Output size cap: keep responses under 5000 words

---

### Fix 3.2: `research.md` Scope Constraints (§8.4)

**File:** `agents/research.md`

**Problem:** The research agent can search any Slack channel and Notion workspace with no scope restrictions.

**Required Changes:**

Add scope constraints:
- Slack: only search channels listed in the task context or the configured pipeline channel. Do not search DM channels or private channels unless explicitly directed.
- Notion: only search databases and pages relevant to the task. Do not browse the entire workspace.
- Do not copy verbatim message content from Slack into artifacts — summarize findings instead.

---

### Fix 3.3: `slack-io.md` Approval Confirmation (§8.5)

**File:** `agents/slack-io.md`

**Problem:** Approval keywords ("approved", "lgtm", "ship it") in casual conversation can false-positive and resume pipeline execution. No confirmation step.

**Required Changes:**

Add a confirmation requirement:
1. When an approval keyword is detected, the agent must verify:
   - The message is a direct reply to the design review thread (not a random conversation)
   - The message author is the task creator or a listed approver
2. If both conditions are met, proceed with approval.
3. If the keyword appears outside a review thread, ignore it.
